\section{Introduction}
\label{sec:introduction}
	It's the end days of homogenous architectures. With the ever-increasing computing demands of High Performance Computing (HPC) applications, nodes in a park are required to incorporate other accelerator beyond the CPU. Not only that, but with the AI boom, many more kinds of accelerators are hitting the HPC ecosystem, evermore increasing its program's complexity.

	One of the answers espoused for this problem are Sequential Task Flow (STF) runtime systems. In those, work is segmented in tasks with explicit dependencies. Those tasks are submitted sequentially and execute asynchronously with respect to their dependencies. With this model it is easy to distribute task between diferent architectures. One could implement a task in OpenMP to run in the CPU and also provide an implementation in CUDA to run in the GPU. The runtime deals with the load balancing and data transfers. Some popular runtimes are PaRSEC \cite{bosilca_parsec_2013} and StarPU \cite{starpu-paper}, which was used in this paper. 
	Both of these systems have a data management side to them, which allows for easier track of dependencies and task submission. StarPU's automatic memory management was essential in enabling many mechanisms for this paper's program optimizations. Many applications adopted this paradime for a pletera of contexts, like dense linear algebra \cite{agullo_achieving_2017, abdulah_accelerating_2022}, Computational Fluid Dynamics (CFD) \cite{couteyen_carpaye_design_2018,leandro_nesi_task-based_2020} and Reverse Time Migration (RTM) applications \cite{boillot_task-based_2014, alonazi_asynchronous_2019}. Important for this paper are the RTM applications.

	RTM is a tool of seismic analysis for detecting subsurface deposits of materials. It propagates a wavefield forwards in time, records it and then integrates the recording back in time to generate a partial image. To build the final result, this process is repeated for many on field recorded waves, each iteration called a \textit{shot}, and the partial images are congealed in the final image. There are no dependencies between \textit{shots}, making RTM applications embarrassingly parallelizable. The Fletcher algorithm \cite{fletcher_reverse_2009} can deal with Tilted Transverse Isotropic (TTI) medium and is the chosen implementation of this paper's program.

	One important characteristic of RTM is that it's recorded data between the forwards and backwards propagation can be massive. A normal volume of $1000 \times 1000 \times 500$ saved in \textit{64bit floating point} results in $~4Gb$ of data, which is saved at manually tuned time steps based on the necessary sample rate of this problem's condition. That makes this kind of application I/O bound. Some research is devoted to lossy methods of reducing the amount of saved memory \cite{huang_towards_2023}, but most focus is in overlapping computation with I/O operations \cite{alonazi_asynchronous_2019, kunas_enhancing_2025}. AlOnazi's work calls back to the STF paradigm, in which the StarPU runtime system was used to manage the I/O and data transfer operations while the computation was done in GPUs using it's out-of-core (OOC) features. This utilization was limited to homogenous computation on GPUs and two level paralelism, where the whole medium was placed in the GPUs shared memory. This resulted in a two tier parallel system, where computations used traditional parallel methods while the I/O used the STF model.
	
	Fundamentally, RTM aplications are stencil aplications, programs originated from the discretization of diferencial equations that flow with time. In this context, STF has been used in first order in many CFD aplications 
	\cite{couteyen_carpaye_design_2018, leandro_nesi_task-based_2020}.
	Nesi's paper directed the implementation of this paper's program, nicknamed \textbf{star-fletcher}. It implements, in the STF model using StarPU, only the forwards migrations step of Fletcher RTM, in which it saves its snapshots to disk. One of Nesi's optimizations, called the Arrow Implementation, follows the concept of time tiling \cite{krishnamoorthy_effective_2007, grosser_hybrid_2018, akbudak_asynchronous_2020, qu_toward_2021, qu_exploiting_2023}, which tiles the problem also considering time as a spacial dimension engendering the reuse of cache. This is achieved by growing a pyramid like shape along the time dimension and then filling the rest to start the process again.
	In adopting this design choice, and the changes it required, the program's I/O communication to disk ended up being completely masked behind its block computation.

	\textbf{Star-fletcher} is the first space-segmented implementation of Fletcher with first tier task parallelism. Therefore, mediums too large to fit inside GPU's memories can now be processed in reasonable time. It also demonstrates the effectiveness of the STF paradigm in masking I/O communications when directing a programs design in to the time tiling space. The evidence of cache reuse has not been tested yeet for this program's current implementation.
